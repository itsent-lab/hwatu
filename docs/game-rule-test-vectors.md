# 공통 게임 규칙 테스트 벡터 운영 기준

## 1. 목적

웹과 Android, iOS·iPadOS, macOS, Windows가 같은 입력에서 같은 점수와 정산 결과를 내는지 자동 검증하기 위한 기준입니다. 규칙 설명은 [`game-rules.md`](game-rules.md), 화면과 타이밍은 [`native-app-implementation-spec.md`](native-app-implementation-spec.md)를 따릅니다.

기계 판정의 기준 파일은 다음 두 개입니다.

- `shared/contracts/game-rule-vectors-v1.json`: 입력과 기대 결과
- `shared/contracts/game-rule-vectors-v1.schema.json`: 파일 구조와 허용 연산

## 2. 계약 구조

최상위 `version`을 바꾸지 않는 변경은 기존 소비자가 이해할 수 있는 필드 추가 또는 사례 추가로 제한합니다. 기존 입력·기대값의 의미가 바뀌면 새 버전 파일을 만듭니다.

각 사례는 다음 필드를 가집니다.

| 필드 | 의미 |
| --- | --- |
| `id` | 버전 안에서 영구적으로 유지할 고유 식별자 |
| `operation` | 실행할 규칙 함수 종류 |
| `description` | 사람이 확인할 한국어 판정 이유 |
| `input` | 플랫폼 중립 입력 JSON |
| `expected` | 실제 결과에서 반드시 일치해야 하는 부분 집합 |

지원 연산은 다음과 같습니다.

| 연산 | 검증 대상 |
| --- | --- |
| `captured-score` | 획득패의 광·열끗·띠·피와 조합 점수 |
| `matgo-settlement` | 맞고 고·박·흔들기·미션·나가리 배수와 지급액 |
| `gostop-settlement` | 고스톱 공통 배수, 패자별 박과 독박 |
| `gostop-balance` | 3인 잔액 이동, 보유액 상한 지급과 컴퓨터 리필 |

`expected`는 부분 일치 계약입니다. 설명용 필드가 플랫폼마다 추가돼도 괜찮지만, 계약에 적힌 값은 모두 같아야 합니다.

## 3. 현재 필수 사례

현재 v1에는 다음 경계값이 포함됩니다.

- 비광 포함 여부가 다른 삼광
- 고도리
- 국진의 열끗·쌍피 전환
- 맞고 기본 금액, 3고, 피박·광박·멍박 중첩
- 고 점수 뒤 배수를 적용하는 순서
- 패자가 획득패 0장일 때 지급 면제
- 고스톱 3고·흔들기·폭탄·멍따·나가리 공통 배수
- 패자별 피박·광박과 고 독박
- 잔액까지만 지급하고 0냥이 된 컴퓨터만 자동 리필

새 규칙이나 버그 수정에는 최소한 정상 사례, 경계 사례, 적용되지 않아야 하는 반례를 추가합니다. 카드 순서가 결과에 영향을 주는 규칙은 순서를 바꾼 사례도 둡니다.

## 4. 플랫폼 적용

각 앱은 배포 빌드에서 사용하는 실제 규칙 모듈에 JSON 벡터를 주입하는 단위 테스트를 둡니다. 테스트 전용으로 규칙을 다시 구현하면 안 됩니다.

1. JSON을 스키마로 검사합니다.
2. `operation`에 따라 실제 규칙 함수를 호출합니다.
3. 수치, 불리언, 코드 배열을 정확히 비교합니다.
4. 한 사례라도 실패하면 규칙 호환 빌드로 배포하지 않습니다.

웹 기준 테스트는 `apps/web/src/test/gameRuleContractVectors.test.ts`입니다. 네이티브 앱은 같은 JSON 원본을 테스트 리소스로 포함하되 앱 번들에 넣을 필요는 없습니다.

## 5. 변경 절차

1. 먼저 `game-rules.md`에서 의도한 규칙을 확정합니다.
2. 기존 사례를 몰래 고치지 말고, 변경 이유가 드러나는 사례를 추가하거나 계약 버전을 올립니다.
3. 웹과 모든 활성 네이티브 구현의 테스트를 실행합니다.
4. 서버가 정산을 재검증하는 항목이면 서버 검증도 함께 맞춥니다.
5. 릴리스 노트에 영향받는 저장 상태 버전과 이전 앱 호환 여부를 기록합니다.

플랫폼 하나만 다른 결과가 필요하다면 공통 벡터를 분기하지 않습니다. 먼저 제품 규칙인지 UI 표현 차이인지 결정하고, UI 차이면 이 계약 밖의 플랫폼 문서에 기록합니다.
