# 네이티브 상태 저장·동기화·오프라인 명세

## 1. 목적

앱 종료, 네트워크 단절, 사용자 전환과 중복 요청에서도 게임 진행과 게임머니를 안전하게 유지하기 위한 기준입니다. 규칙 상태의 필드 의미는 웹 엔진과 계약 벡터를 따르며, 이 문서는 저장 책임과 복구 순서를 정의합니다.

## 2. 저장 영역 분리

| 영역 | 데이터 | 보호·수명 |
| --- | --- | --- |
| 보안 저장소 | 세션 쿠키와 향후 인증 토큰 | 운영체제 키체인/자격 증명 저장소, 로그·백업 제외 |
| 사용자 DB | 맞고 진행 판, 동기화 상태, 고스톱 미정산 큐와 잔액 스냅숏 | 사용자 ID별 분리, 원자적 쓰기 |
| 일반 설정 | AI 난이도, 점당 금액, 버림패 확인, 음량·BGM | 앱 설치 단위; 계정 비밀 금지 |
| 임시 메모리/캐시 | 현재 화면, 디코딩 카드·오디오, 프로필 이미지 | 로그아웃·메모리 압박 때 제거 가능 |

웹 저장 키 이름을 네이티브 스키마 이름으로 그대로 사용할 필요는 없지만 의미와 기본값은 유지합니다.

| 항목 | 웹 기준 | 기본값/제한 |
| --- | --- | --- |
| 맞고 AI | `nsj-hwatu-ai-difficulty-v1` | `normal` |
| 고스톱 AI | `nsj-hwatu-gostop-ai-difficulty-v1` | `normal` |
| 맞고·고스톱 점당 금액 | 서로 다른 설정 키 | 각각 100냥 |
| 맞고 두 번 터치 확인 | `nsj-hwatu-discard-confirmation-v1` | 꺼짐 |
| 오디오 | `nsj-hwatu-audio-v1` | 음소거 꺼짐, 0.28, BGM 꺼짐 |
| 고스톱 미정산 | 사용자별 큐 | 최근 20개, UUID 중복 제거 |
| 고스톱 잔액 | 사용자별 스냅숏 | 각 값 0~999,999,999,999 |

## 3. 공통 레코드 규칙

지속 레코드는 최소한 `schemaVersion`, `userId`, `updatedAt`을 가집니다. 게임 레코드는 `gameUuid`, `stateVersion`, `turnNumber`, `createdAt`, `pendingSync`도 가집니다.

- 날짜는 UTC ISO-8601로 저장합니다.
- 금액과 점수는 부동소수점이 아닌 정수입니다.
- UUID는 정산 중복 방지 키이므로 새 판 생성 때만 바꿉니다.
- 저장은 임시 레코드 작성 후 교체 또는 DB 트랜잭션으로 원자적으로 수행합니다.
- 디코딩·검증에 실패한 레코드는 덮어쓰지 말고 격리 또는 진단 후 새 상태를 만듭니다.
- 다른 사용자 ID의 데이터는 현재 화면 복구 후보가 될 수 없습니다.

## 4. 맞고 저장과 병합

현재 서버는 맞고 저장 상태 v1 이상을 메타 수준에서 받지만, 웹 복원 검증은 상태 버전 2와 3을 지원하고 새 저장은 v3을 사용합니다. v2는 48장, v3은 보너스 포함 50장의 카드 유일성을 기대합니다. 네이티브는 지원하지 않는 미래 버전을 억지로 열지 않습니다.

의미 있는 상태 변경마다 다음 순서를 사용합니다.

1. 규칙 함수가 새 불변 상태를 반환합니다.
2. 로컬 DB에 `pendingSync=true`로 먼저 커밋합니다.
3. UI는 로컬 커밋 성공 후 새 상태를 표시합니다.
4. 로그인·온라인이면 `PUT /api/games/matgo`를 보냅니다.
5. 성공 응답의 잔액과 정산 적용 결과를 반영하고 같은 턴을 `pendingSync=false`로 갱신합니다.
6. 실패하면 로컬 진행을 유지하고 연결 복구 때 최신 상태를 다시 보냅니다.

로컬과 서버 후보 선택은 다음 순서입니다.

- 같은 `gameUuid`: 더 큰 `turnNumber` 우선
- 같은 UUID와 턴: `updatedAt`이 늦은 쪽 우선; 내용이 다르면 진단 이벤트 기록
- 다른 UUID: `createdAt`이 늦은 진행 판 우선
- 종료 판과 진행 판이 충돌: 서버의 정산 적용 응답을 보존하고, 이미 끝난 판을 새 진행 판처럼 복원하지 않음

서버 요청 최대 상태 크기는 UTF-8 524,288바이트입니다. 전송 전에 측정해 넘으면 재시도를 반복하지 않고 진단 오류로 처리합니다.

## 5. 고스톱 정산 큐

고스톱 진행 판은 현재 이어하기 대상이 아닙니다. 신뢰성 대상은 판 종료 정산입니다.

1. 판 종료 시 3인 점수 증감 합이 0인지 확인합니다.
2. 현재 잔액 범위에서 낙관적 잔액 스냅숏을 계산해 한 트랜잭션으로 저장합니다.
3. 정산 요청을 `gameUuid`로 사용자별 보류 큐에 넣습니다. 동일 UUID가 있으면 교체합니다.
4. 서버 요청을 보냅니다.
5. 성공한 UUID만 큐에서 제거하고 서버 반환 잔액으로 스냅숏을 교체합니다.
6. 실패하면 큐를 유지합니다.

앱 시작, 고스톱 로비 진입, 고스톱 게임 진입, 네트워크 복구 시 오래된 항목부터 순차 재전송합니다. 하나가 실패하면 뒤 항목을 보내지 않습니다. 여러 동시 플러시를 금지하고 사용자별 단일 실행 잠금을 둡니다.

큐가 20개를 넘으면 가장 오래된 항목을 조용히 버리는 구현을 피해야 합니다. 현재 웹은 최근 20개만 유지하므로 네이티브는 20개 도달 시 진단·사용자 경고와 즉시 동기화를 우선하고, 데이터 유실 가능 상태를 릴리스 차단 결함으로 취급합니다.

## 6. 오프라인 기능 행렬

| 기능 | 오프라인 동작 |
| --- | --- |
| 앱 실행·로그인 | 새 로그인 불가; 유효 세션과 로컬 프로필이 있어도 서버 확인 전 관리자 기능 금지 |
| 맞고 진행 | 로컬 상태로 계속 가능, 동기화 대기 표시 |
| 맞고 새 판 | 로컬 잔액이 0보다 크고 마지막 서버 상태와 충돌 위험이 없을 때만 허용 |
| 맞고 종료 정산 | 로컬 종료 상태 저장 후 서버 재연결 때 전송 |
| 고스톱 진행 | 현재 방은 계속 가능하지만 앱 종료 후 이어하기 없음 |
| 고스톱 종료 정산 | 큐와 잔액 스냅숏에 저장 후 재전송 |
| 게임머니 리필 | 서버 권한 작업이므로 불가 |
| 회원 관리·프로필 업로드 | 불가; 읽기 캐시는 참고 표시만 가능 |
| 설정·오디오 | 즉시 로컬 반영 |

오프라인 표시는 입력을 가리는 팝업보다 지속적인 상태 배지로 제공합니다. 저장 실패와 서버 동기화 실패는 구분합니다.

## 7. 마이그레이션과 손상 복구

- 앱 DB 버전과 게임 `stateVersion`은 별개로 관리합니다.
- 마이그레이션은 이전 버전 백업 또는 트랜잭션 안에서 수행하고, 성공 뒤에만 버전을 올립니다.
- 각 지원 버전에 대해 정상, 누락 필드, 잘못된 카드 수, 중복 카드, 미래 버전 테스트를 둡니다.
- 미래 버전은 삭제하지 않고 “이 앱 버전에서 열 수 없음”으로 표시합니다.
- 설정 손상은 안전한 기본값으로 복구할 수 있지만, 미정산 큐와 정산된 게임 상태를 자동 삭제하면 안 됩니다.
- 앱 삭제·데이터 초기화가 미정산 데이터를 잃을 수 있다는 점을 진단/지원 문서에 알립니다.

## 8. 완료 테스트

- 카드 선택 직후, 더미 뒤집기 직후, 선택창 표시 중, 판 종료 직후 강제 종료 복구
- 요청은 성공했지만 응답 수신 전 앱 종료된 정산의 중복 방지
- 비행기 모드에서 여러 턴 진행 후 순서대로 동기화
- 사용자 A 로그아웃 후 사용자 B 로그인 때 데이터 격리
- 기기 시간 역행에도 턴 번호가 우선되는 병합
- DB 용량 부족·쓰기 거부 시 게임 중복 진행 방지와 명확한 안내
- 이전 지원 버전에서 현재 버전으로의 실제 마이그레이션
