# 공개 전 점검표

## 한 줄 결론

현재 파일은 운영 정보가 제외된 공개 후보입니다. 다만 운영 설정이 있던 로컬 Git 이력을 그대로 푸시하면 안 됩니다. 이전 작성자 이력은 유지하고, 검증된 `origin/main`을 부모로 하는 새 공개 커밋에 현재 파일만 반영합니다.

## 가장 간단한 절차

1. 저장소와 현재 작업 디렉터리를 백업합니다.
2. `origin/main`을 기준으로 공개용 커밋을 준비합니다. 운영 설정이 있던 현재 로컬 `main`을 부모로 사용하지 않습니다.
3. 다음 명령이 모두 통과하는지 확인합니다.

   ```bash
   npm run check:public
   ```

4. 스테이징된 파일 목록에서 실제 설정, 데이터, 빌드 산출물이 없는지 마지막으로 확인합니다.
5. 검증한 공개 브랜치만 `main`으로 푸시합니다. `git push --all`과 미러 푸시는 사용하지 않습니다.

## 자동으로 검사하는 항목

`npm run check:public`은 다음 항목을 한 번에 검사합니다.

- 공개 후보의 민감한 파일명, 개인키와 대표적인 토큰 패턴
- Git에서 제외된 실제 설정값이 공개 파일에 포함됐는지 여부
- 현재 브랜치 이력에 비공개 운영 파일 경로가 연결됐는지 여부
- 화투패와 공개 음원의 기록된 해시
- Git 작성자 `NSJ <itsent@itsent.co.kr>`
- TypeScript 검사, 웹 테스트, 웹 프로덕션 빌드와 .NET 빌드

의존성 취약점 조회는 최신 외부 데이터베이스가 필요하므로 공개 직전에 npm과 NuGet 검사를 별도로 실행합니다.

## 공개하지 않는 정보

- DB 서버·계정·비밀번호, 연결 문자열, 초기 설정 토큰과 기타 비밀정보
- 실제 도메인, 인증서·개인키, 서버 사용자·그룹과 내부 설치 경로
- 회원 계정, 비밀번호 해시, 프로필 사진, 게임 저장·정산 기록
- DB 덤프, 백업, 로그, 업로드, 내보내기 파일과 운영 전용 음원

이전 커밋의 작성자 이름·이메일과 과거 프로젝트 명칭은 유지할 수 있습니다. 다만 같은 커밋에 실제 운영 설정이나 데이터가 있으면 해당 이력은 공개 브랜치에 연결하지 않습니다.

## GitHub에서 확인할 설정

- secret scanning과 push protection
- Dependabot 경고
- 기본 브랜치 보호
- 필요하지 않은 추가 브랜치·태그·릴리스가 없는지 여부

공개 후에는 새 디렉터리에 다시 복제해 추적 파일, 문서 링크, 라이선스 고지와 빌드를 한 번 더 확인합니다.
