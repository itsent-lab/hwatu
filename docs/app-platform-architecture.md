# 웹·네이티브 앱 저장소 구조

## 기본 원칙

웹앱과 네이티브 앱은 서로 독립된 클라이언트입니다. Android, iOS·iPadOS, macOS와 Windows 앱은 현재 React 웹앱을 WebView로 감싸거나 웹 빌드 결과를 포함하지 않습니다. 각 앱은 운영체제의 네이티브 UI, 저장소, 네트워크, 접근성, 생명주기와 배포 체계를 사용합니다.

웹앱은 계속 지원하며 기존 React 게임의 동작과 배포 방식도 유지합니다.

## 변경할 수 없는 필수 요구사항

다음 항목은 제안이나 권장사항이 아니라 모든 네이티브 앱 구현·검토·출시에서 반드시 지켜야 하는 완료 조건입니다.

1. **100% 네이티브 구현:** Android는 Kotlin·Jetpack Compose, iOS·iPadOS와 macOS는 Swift·SwiftUI/AppKit, Windows는 C#·WinUI로 화면과 동작을 직접 구현합니다.
2. **WebView 절대 금지:** WebView, WebKit 화면, WebView2, 내장 브라우저, 로컬 HTML·JavaScript 번들 및 웹 URL 화면 로딩으로 앱 기능이나 UI를 구현하지 않습니다.
3. **웹 UI/UX 기준 준수:** 화면 계층, 배치, 색상, 크기, 문구, 상태, 팝업, 조작 순서, 애니메이션과 피드백을 웹 기준과 동일하게 구현합니다. 승인되지 않은 차이가 있으면 완료로 판정하지 않습니다.
4. **동일한 오디오 파일 사용:** 웹의 음성·배경음·음향·효과음 파일을 파일명과 바이트가 동일한 상태로 사용하며 재인코딩하거나 플랫폼별 대체음을 만들지 않습니다.
5. **동일한 규칙 결과:** 동일 입력에서 선택 가능 여부, 카드 이동, 점수, 배수, 정산 및 게임 종료 결과가 웹과 일치해야 합니다.
6. **앱 내부 라이선스 고지:** 사용한 자산과 의존성의 라이선스가 표시를 요구하면 해당 앱 안의 접근 가능한 크레딧·라이선스 화면에 표시합니다.

운영체제의 안전 영역, 시스템 권한 창과 같은 플랫폼 제어 영역을 제외한 차이는 자동으로 허용되지 않습니다. 불가피한 차이는 사유와 영향을 문서화하고 별도 승인을 받아야 합니다.

## 디렉터리 구조

```text
hwatu/
├── apps/
│   ├── web/                # 기존 React 19 웹앱과 PWA
│   ├── android/            # Kotlin + Jetpack Compose 네이티브 앱
│   ├── ios/                # Swift + SwiftUI iPhone/iPad 범용 앱
│   ├── macos/              # Swift + SwiftUI/AppKit Mac 앱
│   └── windows/            # C# + WinUI Windows 앱
├── services/
│   └── api/                # ASP.NET Core API, 데이터 접근과 DB 스키마
├── shared/
│   └── contracts/          # 언어 중립 API·게임 규칙 계약과 테스트 벡터
├── docs/                   # 제품 목적, 규칙과 아키텍처 문서
├── deploy/                 # 서버 운영 설정
└── package.json            # 루트 개발 명령을 연결하는 워크스페이스 진입점
```

## 책임 경계

### 웹

`apps/web/`는 현재의 TypeScript 게임 엔진, React 화면, 브라우저 저장소, Web Audio, PWA 자산과 Vitest 테스트를 모두 소유합니다. 네이티브 앱을 위해 웹 코드에 플랫폼 조건문을 추가하지 않습니다.

웹 프로덕션 빌드는 `services/api/wwwroot/`에 생성되어 기존 ASP.NET Core 서버가 정적 파일로 제공합니다.

### Android

`apps/android/`는 Kotlin과 Jetpack Compose로 독립 구현합니다. Android 생명주기, Room 또는 DataStore, 네트워크 클라이언트, 접근성, 백 버튼과 Play 배포 설정을 이 앱에서 관리합니다.

### iPhone·iPad

`apps/ios/`는 Swift와 SwiftUI 기반의 범용 iOS 앱입니다. iPhone과 iPad는 하나의 Xcode 프로젝트에서 공통 도메인 로직을 사용하되 화면 크기와 입력 방식에 맞는 네이티브 레이아웃을 제공합니다.

### macOS

`apps/macos/`는 SwiftUI를 기본으로 하고 필요한 경우 AppKit을 사용하는 별도 Mac 앱입니다. iOS 프로젝트와 소스 파일을 직접 참조하지 않으며, Apple 플랫폼 사이의 실제 중복이 확인되면 별도 Swift Package 분리를 검토합니다.

### Windows

`apps/windows/`는 C#과 WinUI 기반의 별도 Windows 앱입니다. 솔루션, application identity, 설치 패키지와 코드 서명을 자체 관리합니다.

### 서버와 공통 계약

`services/api/`는 모든 클라이언트가 사용하는 서버입니다. 현재 웹의 쿠키·CSRF 인증은 그대로 유지합니다. 네이티브 앱 인증은 앱 프로젝트를 시작할 때 토큰 저장, 갱신, 폐기와 기기 보안을 포함한 별도 API 설계를 확정해야 하며 웹 쿠키를 그대로 재사용한다고 가정하지 않습니다.

서로 다른 언어 사이에는 구현 코드를 공유하지 않습니다. `shared/contracts/`에는 API 스키마, 오류 코드, 규칙 테스트 벡터와 저장 데이터 버전처럼 언어에 독립적인 계약만 둡니다. 현재 TypeScript 게임 엔진은 웹 구현이며 네이티브 앱의 직접 의존성이 아닙니다.

### 현재 서버의 역할

현재 서버의 제품 책임은 **회원 관리와 게임 기록 관리**로 한정합니다.

- 회원가입, 로그인, 인증, 사용자·관리자 계정과 프로필을 관리합니다.
- 게임 저장 데이터와 맞고·고스톱 정산 기록을 저장하고 조회합니다.
- 현재 게임 진행, 규칙 판정, AI 행동과 화면 상태는 각 클라이언트에서 처리합니다.
- ASP.NET Core가 웹 빌드 결과를 정적 파일로 제공하는 것은 배포 구성에 해당하며, 서버가 게임 진행을 담당한다는 의미는 아닙니다.

서버에 새로운 게임 진행 책임을 추가할 때는 회원 또는 기록 관리에 필요한 기능인지 먼저 확인합니다. 해당 범위를 벗어나는 기능은 별도 요구사항과 설계 결정 없이 현재 서버 기능으로 간주하거나 구현하지 않습니다.

### 대전 기능의 상태

사용자 간 대전은 **향후 개발 계획**이며 현재는 구현되어 있지 않습니다. 따라서 현재 API, 데이터베이스, 클라이언트와 공통 계약에 대전 기능이 존재한다고 가정하지 않습니다.

대전 개발을 시작할 때는 매칭과 방의 수명주기, 실시간 상태 동기화, 규칙 판정 주체, 통신 장애와 재접속, 이탈 처리, 결과 확정과 기록 저장, 부정행위 방지 범위를 먼저 설계해야 합니다. 이 항목들은 검토 대상이며 현재 확정된 구현 사양이 아닙니다. 설계가 확정되기 전에는 미래 대전을 예상한 임시 API, 데이터베이스 테이블 또는 클라이언트 분기를 추가하지 않습니다.

### 웹을 기준으로 한 화면 구성

웹앱의 화면 UI를 모든 네이티브 앱의 기준 화면으로 사용합니다. 네이티브 앱은 웹 화면을 WebView로 표시하거나 React 컴포넌트와 CSS를 직접 참조하지 않고, 대응하는 화면 크기의 웹 UI를 각 운영체제의 네이티브 UI 기술로 동일하게 다시 구현합니다. “네이티브 구현”은 구현 기술이 플랫폼별이라는 의미이며 앱의 디자인을 플랫폼별로 다시 해석하거나 바꾼다는 의미가 아닙니다.

WebView 금지는 예외 없는 필수 조건입니다. `WKWebView`·`UIWebView`, Android WebView, WinUI WebView2를 비롯한 웹 렌더링 컴포넌트, 내장 브라우저 화면, 로컬 HTML·JavaScript 번들 또는 웹앱 원격 URL을 앱 화면으로 사용하는 구현은 허용하지 않습니다. 도움말이나 라이선스의 외부 링크는 운영체제 기본 브라우저로 열 수 있지만, 앱의 기능과 화면을 대신 렌더링하는 용도로 사용하지 않습니다.

다음 항목은 웹과 동일하게 유지합니다.

- 화면 계층, 섹션 구성, 정보 배치와 정렬
- 색상, 여백, 크기 비율, 모서리, 테두리, 아이콘과 카드 표현
- 문구, 버튼, 선택 항목과 사용자가 수행할 수 있는 행동
- 화면 상태, 오버레이, 확인 창, 전환 순서와 애니메이션 시점
- 점수, 정산, 게임 종료 결과, 중요한 선언과 효과 발생 시점

화면 크기와 DPI가 다르면 웹의 반응형 레이아웃에서 같은 조건에 해당하는 구성을 기준으로 크기만 적응시킵니다. 안전 영역, 시스템 권한 창, 상태 표시줄, 입력 장치와 접근성 API처럼 운영체제가 직접 제어하는 부분만 예외로 하며, 이를 이유로 앱 고유 UI를 임의 변경하지 않습니다. 웹과 다르게 구현해야 하는 불가피한 항목은 차이, 사유와 영향을 문서화하고 별도 승인을 받은 뒤 반영합니다.

각 화면은 동일한 데이터와 상태를 사용해 웹 기준 스크린샷과 네이티브 앱 스크린샷을 대응 화면 크기별로 비교합니다. 주요 게임 상태, 팝업, 오류, 로딩, 빈 화면과 접근성 확대 상태까지 비교하고 승인되지 않은 시각적·동작 차이가 있으면 완료로 처리하지 않습니다.

### 웹 오디오·효과 파일 재사용

웹에서 실제 사용하는 선언 음성, 배경음, 일반 음향과 게임 효과음 파일은 다른 플랫폼에서도 모두 동일한 파일을 그대로 사용합니다. 배경음의 게임 상황별 재생 속도·음량 전환도 같은 단계와 값을 따릅니다. 플랫폼별로 대사, 음색, 음질, 속도, 길이 또는 효과가 달라지는 별도 오디오 파일을 임의로 만들거나 다른 소리로 대체하지 않습니다.

네이티브 앱 패키징을 위해 파일을 각 앱의 자산 디렉터리에 복사할 수 있지만 파일명과 바이트를 유지하고, 재인코딩·편집·정규화하지 않으며 웹 원본과 동일한 파일인지 해시로 확인합니다. 재생 API와 음량 제어는 플랫폼별 네이티브 방식으로 구현할 수 있지만 어떤 상황에 어떤 파일을 재생하는지와 게임 효과가 발생하는 시점은 웹을 기준으로 유지합니다.

특정 플랫폼이 현재 파일 형식을 지원하지 않는 경우 그 플랫폼만을 위한 변환본을 만들지 않습니다. 모든 대상 플랫폼에서 사용할 공통 원본 형식을 검토하고 웹을 포함한 전체 자산 기준 변경을 별도 승인받습니다. 파일의 출처, 라이선스와 AI 생성 고지는 [`apps/web/public/audio/ATTRIBUTION.txt`](../apps/web/public/audio/ATTRIBUTION.txt) 및 [`reference-and-asset-policy.md`](reference-and-asset-policy.md)를 따르며, 네이티브 앱에서도 필요한 고지를 유지합니다.

### 네이티브 UI/UX·효과 완료 확인표

아래 항목은 단순히 화면에 요소가 있거나 한 번 재생되는지만 확인하지 않습니다. 웹과 같은 상태에서 발생 조건, 표시 위치, 지속 시간, 겹침 우선순위, 사용자 입력 차단 범위, 종료 후 상태와 음향 시점까지 같아야 완료로 판정합니다. 하나라도 확인되지 않은 플랫폼은 `100% 동일` 또는 `완료`로 표시하지 않습니다.

- [ ] 홈, 로그인, 서버 설정, 로비, 선 정하기, 맞고, 고스톱, 결과, 관리자와 법적 고지 화면의 구조·좌표·크기 비율·문구를 대응 화면 크기에서 비교했습니다.
- [ ] 기본, 로딩, 빈 상태, 연결 오류, 인증 만료, 선택, 비활성, 진행 중과 완료 상태를 모두 비교했습니다.
- [ ] 패 나누기가 모드별 장수와 웹의 순서·간격·지속 시간으로 재생되고 완료 전 잘못된 입력을 막습니다.
- [ ] 손패 내기, 더미 뒤집기, 바닥 이동, 획득패 이동과 점수 변화가 실제 게임 상태 변경과 함께 애니메이션됩니다.
- [ ] AI 생각 중 표시가 난이도별 행동 지연과 동기화되고 AI 행동이 끝나면 남지 않습니다.
- [ ] 보너스패, 피 뺏기, 뻑·연속뻑·삼연뻑, 쪽, 따닥, 싹쓸이, 자뻑, 싼 패 먹기, 폭탄, 흔들기, 총통과 국진 전환의 선언 효과를 각각 검증했습니다.
- [ ] 고·스톱 선택, 고 횟수 증가, 승리·패배·나가리와 최종 정산 효과가 올바른 결과와 함께 표시됩니다.
- [ ] 연속 효과의 우선순위와 중첩, 같은 효과의 연속 재발생, 효과 중 팝업 표시와 입력 가능 여부가 웹과 같습니다.
- [ ] 효과음, 플레이어 음성, 상대별 음성, 배경음, 음소거와 음량 변경이 웹과 동일한 원본 파일 및 재생 시점을 사용합니다.
- [ ] 자동 치기, 무르기, 나가기 예약, 새 판과 백그라운드 전환 시 애니메이션·음향·진행 작업이 올바르게 시작·취소·복구됩니다.
- [ ] 동작 줄이기, 키보드 접근, 화면 읽기와 포커스 순서에서도 기능이 누락되지 않고 시스템 설정에 맞게 효과가 조정됩니다.
- [ ] 화면 비교, 규칙 벡터, 오디오 파일 무결성, API 통합, 디버그 테스트와 릴리스 빌드 검증 결과를 기록했습니다.

### 게임 규칙 일치

모든 플랫폼은 [`game-rules.md`](game-rules.md)에 정리된 맞고·고스톱 규칙을 동일하게 따릅니다. 웹의 현재 TypeScript 구현은 동작 기준이지만 다른 플랫폼에서 직접 불러오는 공용 라이브러리는 아닙니다.

플랫폼별 구현은 동일한 입력에서 같은 선택 가능 여부, 카드 이동, 점수, 배수, 정산과 종료 결과를 내야 합니다. 이를 검증하기 위해 `shared/contracts/`의 언어 중립 규칙 테스트 벡터를 사용하고, 규칙을 변경할 때는 기준 문서와 테스트 벡터를 먼저 또는 함께 갱신한 뒤 각 플랫폼 테스트에 반영합니다.

### 개발 전 확인표

새 플랫폼이나 이후 대전 기능을 개발할 때 다음 순서로 범위를 확인합니다.

1. 작업이 현재 기능인지 향후 계획인지 구분합니다.
2. 서버 변경이 회원·기록 책임 안에 있는지 확인합니다.
3. 화면 구조, 배치, 스타일, 상태와 사용자 흐름을 현재 웹 구현 및 기준 스크린샷과 대조합니다.
4. 웹에서 사용하는 동일한 음성·배경음·음향·효과음 파일과 자산 고지를 반영합니다.
5. `game-rules.md`와 공통 테스트 벡터로 규칙 결과가 일치하는지 검증합니다.
6. [`reference-and-asset-policy.md`](reference-and-asset-policy.md)에 따라 필요한 라이선스 정보를 앱 내부에 표시합니다.
7. 대전 작업이라면 미확정 설계 항목을 먼저 결정하고 별도 승인된 범위에서 구현합니다.

## 현재 루트 명령

저장소 루트에서 기존 명령을 계속 사용할 수 있습니다.

```bash
npm run dev
npm run dev:server
npm run typecheck
npm test
npm run build
dotnet build services/api/Hwatu.Server.csproj
```

루트 npm 명령은 `apps/web/` 워크스페이스로 전달됩니다. Android, iOS, macOS와 Windows의 공식 빌드 명령은 각 네이티브 프로젝트를 실제로 생성한 뒤 해당 디렉터리에 추가합니다.

## 네이티브 앱 개발 계획

네이티브 앱은 플랫폼별 프로젝트를 한꺼번에 생성한 뒤 기능을 흩어 구현하지 않고 다음 단계를 순서대로 진행합니다. 먼저 한 플랫폼을 기준 구현으로 선정해 전체 흐름과 공통 계약을 검증한 뒤, 확인된 구조를 바탕으로 나머지 플랫폼을 독립 구현합니다. 기준 플랫폼은 개발을 시작할 때 지원 기기와 배포 우선순위에 따라 확정하며 문서만으로 임의 지정하지 않습니다. 각 단계의 검증을 통과한 뒤 다음 단계로 이동합니다.

### 1단계: 웹 기준선과 공통 계약 확정

- 웹의 화면 목록, 사용자 흐름, 규칙, AI 행동, 저장 형식과 API 요청·응답을 목록화합니다.
- 대응 화면 크기별 웹 기준 스크린샷과 화면 상태 목록을 준비합니다.
- [`game-rules.md`](game-rules.md)를 기준으로 플랫폼 공통 규칙 테스트 벡터를 `shared/contracts/`에 정의합니다.
- 고정된 카드 배열과 난수 시드를 사용하는 재현 가능한 게임 시나리오를 정의합니다.
- 웹에서 사용하는 음성·배경음·음향·효과음 파일 목록과 해시, 재생 시점 및 라이선스 고지를 확정합니다.
- 현재 서버 범위가 회원과 기록 관리이며 대전은 미구현임을 API 설계 기준으로 확인합니다.

### 2단계: 플랫폼 프로젝트 기반 구성

- Android, iOS·iPadOS, macOS와 Windows의 영구 앱 식별자, 최소 지원 버전, 서명 방식과 배포 계정을 확정합니다.
- 각 플랫폼의 네이티브 UI, 보안 저장소, 네트워크, 오디오와 생명주기 구조를 독립적으로 구성합니다.
- 도입할 의존성의 유지보수 상태와 라이선스를 확인하고 앱 내부 고지 필요 여부를 기록합니다.
- 임시 앱 식별자나 개발용 인증 정보를 출시 설정으로 재사용하지 않습니다.

### 3단계: 규칙과 게임 상태 구현

- UI보다 먼저 카드, 규칙, 점수, AI, 정산과 게임 상태를 플랫폼별 도메인 계층으로 구현합니다.
- 웹 TypeScript 코드를 직접 참조하거나 다른 네이티브 플랫폼 구현을 복사해 의존하지 않습니다.
- 모든 공통 규칙 테스트 벡터를 통과하고 웹과 동일한 입력에서 같은 결과가 나오는지 비교합니다.
- 저장 데이터에 명시적인 버전을 두고 이전 버전을 읽기 위한 마이그레이션 테스트를 작성합니다.

### 4단계: 화면·오디오·접근성 구현

- 웹의 화면 구조, 배치, 색상, 크기 비율, 문구, 상태, 행동 순서와 확인 흐름을 동일하게 네이티브 UI로 구현합니다.
- 대응 화면 크기와 동일 상태에서 웹·네이티브 스크린샷을 비교하고 승인되지 않은 차이를 수정합니다.
- 웹과 동일한 모든 오디오 파일을 재인코딩 없이 포함하고 해시와 재생 시점을 검증합니다.
- 큰 글자, 충분한 터치 영역, 색 대비, 화면 읽기, 키보드·리모컨 입력과 동작 줄이기 등 적용 가능한 플랫폼 접근성을 반영합니다.
- 화면 잠금, 전화·알림, 오디오 포커스 변경과 백그라운드 전환 이후 게임과 오디오가 올바르게 복구되는지 확인합니다.

### 5단계: 회원·기록 서버 연동

- 네이티브 앱용 인증 방식과 토큰 발급·갱신·폐기 절차를 확정한 뒤 플랫폼 보안 저장소에 자격 증명을 보관합니다.
- 웹 쿠키·CSRF 인증을 네이티브 앱에 그대로 적용하거나 비밀번호·토큰·운영 비밀을 앱에 하드코딩하지 않습니다.
- 네트워크 끊김, 시간 초과, 재시도와 앱 강제 종료가 발생해도 기록이 유실되거나 중복 저장되지 않도록 검증합니다.
- 서버 API와 저장 데이터 버전의 호환성을 확인하고 회원·기록 이외의 서버 책임을 임의로 추가하지 않습니다.

### 6단계: 플랫폼별 출시 검증

- 시뮬레이터·에뮬레이터 검사와 별도로 지원 대상 실제 기기에서 전체 게임 흐름을 검증합니다.
- 신규 설치, 업데이트, 로그아웃, 자격 증명 만료, 오프라인 복귀와 기존 기록 마이그레이션을 확인합니다.
- 개인정보 안내, 요청 권한, 앱 서명, 배포 설정과 스토어 제출 정보를 검토합니다.
- 출시 직전에 각 앱 스토어와 운영체제의 최신 정책을 확인하고 이전에 확인한 정책을 그대로 가정하지 않습니다.
- [`reference-and-asset-policy.md`](reference-and-asset-policy.md)에 따라 자산과 의존성의 필수 고지를 앱 내부에서 확인합니다.

### 후속 단계: 사용자 간 대전

대전은 위 네이티브 앱의 기본 기능과 별도의 향후 개발 단계입니다. 매칭, 방 관리, 실시간 동기화, 규칙 판정 주체, 재접속, 이탈 처리, 결과 확정과 부정행위 방지 설계를 먼저 승인한 뒤 서버·클라이언트 작업 범위를 정합니다.

## 네이티브 개발 필수 주의 사항

### 아키텍처와 동작 일치

- WebView로 웹앱을 감싸거나 웹 빌드 결과를 네이티브 앱에 포함하는 행위는 절대 금지하며 예외를 두지 않습니다.
- 코드 검토와 출시 검증에서 `WKWebView`·`UIWebView`, Android WebView, WebView2, WebKit 기반 화면, HTML·JavaScript 앱 번들 및 웹 URL 화면 로딩 참조가 없는지 검색합니다.
- 플랫폼 간 UI·저장소 구현 코드를 직접 참조하지 않고 언어 중립 계약과 테스트 벡터만 공유합니다.
- UI 구현 기술은 플랫폼별 네이티브 방식을 사용하되 화면 디자인과 동작은 웹을 동일하게 재현합니다.
- 운영체제 제약으로 차이가 불가피하면 임의 반영하지 않고 차이와 사유를 문서화해 별도 승인을 받습니다.
- 미래 대전을 예상한 임시 API, 데이터베이스 테이블이나 조건 분기를 현재 구현에 추가하지 않습니다.

### 인증과 보안

- 네이티브 인증은 웹 인증과 분리해 설계하고 토큰은 각 운영체제의 보안 자격 증명 저장소에 보관합니다.
- 비밀번호, 개인키, API 비밀키, 운영 서버 자격 증명과 로컬 개발 설정을 소스나 앱 패키지에 포함하지 않습니다.
- 로그, 오류 보고와 분석 데이터에 인증 정보, 회원 개인정보 또는 게임 기록의 민감한 값이 남지 않게 합니다.
- 통신 실패와 인증 만료를 구분하고 토큰 갱신 실패 시 안전하게 로그아웃하거나 재인증을 요청합니다.

### 데이터와 생명주기

- 저장 형식에 버전을 두고 업데이트 전·후 데이터 호환성과 실패 시 복구 경로를 검증합니다.
- 데이터 마이그레이션은 원본을 즉시 덮어쓰거나 삭제하지 않고 실패해도 이전 데이터로 복구할 수 있게 설계합니다.
- 저장과 서버 전송은 중복 실행되어도 같은 기록이 여러 번 생성되지 않도록 설계합니다.
- 앱 종료, 프로세스 제거, 화면 회전, 창 크기 변경, 백그라운드 전환과 기기 재시작 상황을 검증합니다.
- 중단된 게임을 복원할 때 규칙 상태, 차례, 카드 위치, 점수와 오디오 상태가 서로 어긋나지 않게 합니다.

### 재현성·시간·호환성

- 카드 섞기와 AI 판단에 사용하는 난수 시드를 기록해 동일한 문제 상황을 테스트에서 재현할 수 있게 합니다.
- 날짜와 기록 시각은 저장·통신 기준과 사용자 표시 시간대를 구분하고 플랫폼마다 다른 시각으로 기록되지 않게 합니다.
- 통화, 게임머니와 숫자 표시는 반올림, 자릿수와 지역화 기준을 명시해 플랫폼별 결과 차이를 방지합니다.
