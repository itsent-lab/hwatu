# 네이티브 플랫폼 차이·배포·릴리스 기준

## 1. 목적

공통 제품 동작은 같게 유지하면서 Android, iOS·iPadOS, macOS, Windows의 저장소, 창, 입력, 권한과 배포 차이를 명시합니다. 이 문서는 구현 완료 선언과 스토어 제출 전 최종 점검 기준입니다.

## 2. 공통 구현 경계

- 네이티브 앱은 WebView 셸이 아니며 각 플랫폼 UI·저장소 코드를 직접 사용합니다.
- 다른 플랫폼의 UI·저장소 구현을 소스 수준으로 직접 참조하지 않습니다.
- 공유 가능한 것은 `shared/contracts/`의 JSON 계약, 카드·오디오 원본과 검증 자료입니다.
- 규칙을 언어별로 구현하더라도 공통 벡터를 모두 통과해야 합니다.
- 서버 API를 추측해 확장하지 않고 지원 클라이언트·최소 버전 계약을 먼저 추가합니다.

## 3. 플랫폼별 필수 차이

| 항목 | Android | iOS·iPadOS | macOS | Windows |
| --- | --- | --- | --- | --- |
| 보안 저장 | Keystore 기반 암호화 저장 | Keychain | Keychain | Credential Locker/DPAPI |
| 로컬 DB | Room/SQLite 계열 | Core Data/SQLite 계열 | SwiftData/Core Data/SQLite 계열 | SQLite/앱 데이터 계열 |
| 뒤로가기 | 시스템 Back과 예측 뒤로가기; 진행 판 확인 | 내비게이션/스와이프가 진행 판을 버리지 않음 | 창 닫기와 앱 종료 구분 | 창 닫기와 시스템 Back/키보드 구분 |
| 창 | 폰·태블릿 회전/멀티윈도 | 폰·태블릿, Split View | 자유 크기·전체 화면 | 자유 크기, DPI·다중 모니터 |
| 사진 | 시스템 Photo Picker 우선 | PhotosPicker 우선 | NSOpenPanel/PhotosPicker | FileOpenPicker |
| 입력 | 터치·키보드 | 터치·키보드·포인터 | 마우스·키보드·트랙패드 | 마우스·키보드·터치·펜 |
| 업데이트 | Play 배포 정책 | App Store 정책 | App Store 또는 서명·공증 경로 확정 | Store/MSIX 또는 서명 설치 경로 확정 |

사진은 플랫폼 선택기에서 읽은 뒤 앱 내부에서 중앙 정사각형 512×512 JPEG로 변환합니다. 원본 전체 사진이나 위치 메타데이터를 서버에 보내지 않습니다.

## 4. 앱 생명주기

- 비활성 전환 전에 진행 중인 의미 있는 맞고 상태를 원자적으로 저장합니다.
- 백그라운드에서 긴 AI 계산·타이머·오디오를 계속하지 않습니다.
- 복귀 시 저장 상태와 화면 상태를 한 번 재조정하고 같은 AI 행동을 중복 예약하지 않습니다.
- 창 닫기, 시스템 뒤로가기, 스와이프 뒤로가기에서 `예약 나가기`, `바로 나가기`, `계속` 의미를 보존합니다.
- 운영체제가 앱을 즉시 종료해도 마지막 완료된 트랜잭션으로 복구돼야 합니다.

## 5. 버전과 호환성

각 릴리스는 앱 버전, 빌드 번호, API 계약 버전, 로컬 DB 버전, 게임 상태 버전을 기록합니다. 서버 호환 응답의 최소 버전보다 낮으면 인증·게임 요청을 진행하지 않습니다.

Android, iOS·iPadOS, Windows는 현재 `/api/client/status` 계약에 허용 식별자가 없습니다. 다음 항목이 서버와 `shared/contracts/client-api-v*.json`에 추가돼야 출시 가능합니다.

- 안정적인 `X-Hwatu-Client` 값
- 플랫폼별 최소 클라이언트 버전
- 서버 응답의 호환 판정
- 이전 버전 차단·업데이트 안내 테스트

## 6. 빌드 구성과 비밀정보

- 개발·스테이징·운영 API 주소는 빌드 구성으로 분리합니다.
- 운영 서버 주소 외 인증서, 비밀번호, 설정 토큰, 세션 값을 소스·앱 번들·크래시 로그에 넣지 않습니다.
- 디버그 메뉴와 네트워크 상세 로그는 배포 빌드에서 제거합니다.
- 앱 식별자, 서명 팀, 인증서, 프로비저닝, 스토어 계정은 비밀 관리 위치에서 다룹니다.
- 카드·오디오·글꼴 등 번들 자산의 라이선스와 해시를 빌드 산출물 기준으로 검증합니다.

## 7. 테스트 행렬

| 범주 | 필수 검증 |
| --- | --- |
| 규칙 | 공통 JSON 벡터 전체, AI 결정의 유효 수, 정산 합계 |
| API | 정상·401·419·429·5xx·오프라인·느린 응답·중복 정산 |
| 저장 | 강제 종료, 버전 이동, 손상, 용량 부족, 사용자 전환 |
| UI | compact/regular, 세로/가로, 창 최소/최대, 320pt, 고 DPI |
| 접근성 | 큰 글자, 스크린 리더, 키보드 전용, 동작 줄이기, 대비 |
| 오디오 | 음소거, BGM, 백그라운드, 오디오 중단, 자산 누락 |
| 성능 | 첫 화면, 패 돌리기, 50장 렌더, 메모리 경고, 저사양 기기 |
| 보안 | HTTPS, 쿠키 보호, 로그 비밀 누출, 사진 메타데이터, 세션 삭제 |

최소 지원 OS와 실제 시험 기기 목록은 각 플랫폼 README에서 버전별로 유지합니다. 시뮬레이터만으로 카메라/사진, 오디오 중단, 메모리 압박과 스토어 서명을 완료 판정하지 않습니다.

## 8. 릴리스 게이트

다음 항목이 모두 충족돼야 후보 빌드를 배포합니다.

- 공통 규칙 벡터, 플랫폼 단위·통합 테스트 통과
- 현재 API 호환성 확인과 최소 앱 버전 정책 통과
- 이전 배포 버전의 DB·게임 상태 마이그레이션 통과
- 로그인, 맞고 한 판, 고스톱 한 판, 오프라인 복구의 실제 기기 스모크 테스트
- 기준 화면과 접근성 상태 검토
- 카드·오디오 자산 누락/해시/라이선스 검증
- 개인정보 처리방침, 크레딧, MIT 라이선스를 앱 안에서 열 수 있음
- 스토어 메타데이터의 연령·가상 게임머니·개인정보 답변 검토
- 크래시·진단 로그에 비밀정보가 없고 지원용 앱/API/상태 버전은 확인 가능
- 롤백 또는 최소 버전 상향 절차와 담당자 확인

## 9. 배포 후 확인과 롤백

배포 뒤에는 설치·업데이트, 로그인, 호환성 응답, 각 게임 진입, 저장·정산, 법적 고지를 실제 배포 채널에서 확인합니다. 서버 오류율, 로그인 제한 증가, 정산 실패·중복, 저장 마이그레이션 오류를 우선 관찰합니다.

치명적 규칙·정산 오류가 있으면 신규 배포 중단, 서버 최소 버전 정책 또는 스토어 긴급 업데이트 중 안전한 경로를 선택합니다. 로컬 DB를 삭제하는 방식으로 복구하지 않으며, 미정산 큐를 보존하는 수정 버전을 제공합니다.

## 10. 플랫폼 README 완료 기록

각 `apps/<platform>/README.md`에는 다음만 플랫폼 사실로 기록합니다.

- 실제 기술 스택과 최소 OS
- 확인된 빌드·테스트 명령
- 앱 식별자와 권한의 공개 가능한 설명
- 구현 완료 화면과 미완료 범위
- 공통 계약 버전과 마지막 검증 날짜
- 스토어/서명 절차의 비밀 없는 개요

추측한 명령이나 아직 지원하지 않는 기능을 완료로 기록하지 않습니다.
